# AWEBO – UX Audit Sprint 1

**Scope:** Public Experience (Landing → Launch Detail → Participation Intent)  
**Sprint:** 1  
**Budget:** $400  
**Focus:** Clarity, Trust, Conversion

---

# 1. Context & Objective

## Sprint Objective

Redesign and optimize the public-facing front-end experience of AWEBO, focusing exclusively on the participant journey:

Landing → Launch Detail Page → Participation Intent.

This sprint does NOT include:

- Creator dashboard
- Token creation flow
- Full launch wizard
- Backend logic redesign

The focus is to:

- Clarify AWEBO’s value proposition
- Establish trust from first contact
- Reduce cognitive friction
- Properly position the phygital differentiator
- Create a clean structural foundation for future sprints

---

# 2. UX Thesis (What We’re Actually Building)

> We are not building a launchpad.  
> We are building public trust around on-chain cultural objects.

In Sprint 1, the product is:

- Clarity
- Trust
- Professional presentation
- Strong visual hierarchy

It is NOT:

- A DeFi terminal
- A trading dashboard
- A hype-driven token board
- A gambling interface

The experience should feel like:

- Cultural curation
- Professional production
- A premium platform

---

# 3. User Reframing (Sprint 1 Scope)

In this sprint, the user is not creating.  
They are:

- Discovering
- Evaluating
- Assessing legitimacy
- Deciding whether to participate

## Job To Be Done

> “Help me quickly understand what AWEBO is, why it’s different, and whether I can trust it enough to participate in a launch.”

## Expected Emotional Arc

| Stage         | Emotion    |
| ------------- | ---------- |
| Arrival       | Curiosity  |
| Understanding | Clarity    |
| Evaluation    | Security   |
| Decision      | Confidence |

If the user feels confusion or overload at any point, the UX fails.

---

# 4. Home / Hero as a Product Surface

The Hero section must answer—without scroll:

1. What is AWEBO?
2. Why is it different?
3. What do I get as a participant?
4. What should I do next?

## Observed Problems

- Messaging may be too technical or abstract
- Phygital value not clearly differentiated
- Weak visual hierarchy
- Competing calls to action
- Lack of immediate clarity

## Sprint 1 Quick Wins

- More declarative headline
- Clear, benefit-driven subheadline
- One primary CTA
- Reduced visual noise
- Stronger hierarchy between sections

## Future Improvements

- Progressive storytelling
- Cultural narrative refinement
- Strategic animation

---

# 5. Anti-Overload Principle

Current risk: Too much exposed at once.

UX Rule for Sprint 1:

> One decision per screen.

Landing → Understand  
Launch Page → Evaluate  
CTA → Decide

This reduces friction and builds confidence progressively.

---

# 6. Launch Detail Page (Primary Sprint Asset)

This is the most important page in Sprint 1.

If this page feels unclear, AWEBO feels untrustworthy.

## Observed Risks

- Tokenomics lack hierarchy
- Merch appears secondary or unclear
- CTA not dominant enough
- Technical information overwhelms narrative
- No emotional framing

## Recommended Structure

1. Project identity (name, visual, positioning)
2. Clear description
3. What’s included (Token + Physical component)
4. Key parameters (supply, price, timeline)
5. Primary CTA

## Sprint 1 Quick Wins

- Clear Digital vs Phygital comparison block
- Clean progress indicator
- Minimal, readable countdown
- Clear CTA states (before / during / after launch)
- Reduced technical clutter

## Future Improvements (Sprint 2+)

- Interactive merch preview
- Participation simulation
- Expandable storytelling sections
- Creator credibility layer

---

# 7. Phygital as Core Differentiator

Currently, phygital may feel like an add-on.

It must feel like:

> The reason AWEBO exists.

## Sprint 1 Enhancements

- Dedicated visual section
- “Phygital Enabled” badge or label
- Clear explanation of eligibility
- Simple comparison with traditional launchpads

## Future Enhancements

- Cultural narrative layer
- Emotional storytelling
- Post-launch lifecycle explanation

---

# 8. What Should Be Removed or Hidden

The public homepage should NOT resemble:

- A trading board
- A leaderboard
- A meme token aggregator
- A live speculation interface

Avoid on Home:

- Trending tokens
- Live charts
- Volume metrics
- Rapid flashing indicators

These can exist deeper in the ecosystem, but not in Sprint 1 public surface.

---

# 9. Navigation Simplification

The navbar should be minimal and intentional:

- Logo
- Explore / Launches
- Create Token (positioned, even if limited)
- Docs
- Connect Wallet (contextual state-based)

Reduce cognitive branching.

---

# 10. Conversion Friction Audit

## Identified Frictions

1. Unclear definition of phygital
2. Uncertainty about safety
3. Confusion about post-purchase flow
4. Wallet interaction ambiguity
5. Overwhelming technical terms

## Sprint 1 Fixes

- “How it works” in 3 steps
- Microcopy under CTA
- Clear post-click state feedback
- Icon-based explanation blocks
- Simplified parameter labeling

---

# 11. Trust Layer Reinforcement

Trust is the invisible product.

To reinforce it:

- Clear contract and network info
- Calm, confident tone
- Eliminate hype language
- Professional typography
- Strong spacing and hierarchy
- Clear system feedback

Perceived legitimacy is a UX outcome.

---

# 12. Implementation Priority

## High Priority (Sprint 1 Must-Ship)

- Navbar simplification

- Increase clarity
- Professionalize presentation
- Strengthen trust
- Prepare structural foundations for Sprint 2
